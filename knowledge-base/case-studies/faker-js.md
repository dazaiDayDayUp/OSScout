# Faker.js 与 colors.js 恶意注入事件 —— 维护者信任崩塌

## 背景

2022 年 1 月 9 日，流行 npm 包 `colors.js`（每周下载量约 2000 万次）和 `faker.js`（每周下载量约 2700 万次）的维护者 Marak Squires 发布了恶意版本：

- `colors.js` v1.4.1：引入无限循环，在控制台打印乱码 "LIBERTY LIBERTY LIBERTY"
- `faker.js` v6.6.6：清空仓库代码，README 改为讽刺性内容

这两个包被数千个项目依赖，包括 Amazon AWS 的 SDK、Jest 测试框架等顶级项目。

## 早期征兆

- **维护者倦怠公开表达**：Marak 在 2020 年 11 月的一条 GitHub issue 中公开表示"大公司使用我的免费工作却没有回报"，情绪信号强烈
- **资助请求无果**：作者多次尝试通过 GitHub Sponsors 获得资助，但收效甚微
- **单点维护**：faker.js 长期只有 Marak 一个核心维护者
- **社区_fork 尝试**：部分社区成员曾提议 fork 项目，但被维护者阻止

## 影响

- 全球 CI/CD 构建大面积失败
- Jest 等测试框架被迫紧急发布补丁版本锁定 colors.js 版本
- AWS SDK 团队公开回应受影响
- 引发了关于"开源可持续性"的广泛讨论
- faker.js 社区最终 fork 为 `@faker-js/faker`，原项目废弃

## 教训

1. **维护者经济激励缺失是高风险**：长期无回报的开源维护会导致倦怠和报复性行为
2. **公开情绪表达是强预警信号**：维护者在社交媒体/issue 中的抱怨、疲惫表达需要被监控
3. **社区 fork 意愿反映治理危机**：当社区开始认真讨论 fork 时，原项目的信任已经严重受损
4. **版本锁定策略**：生产环境应锁定依赖版本，避免自动接受 patch 更新
5. **供应链安全**：需要监控维护者账户安全（是否被盗号）和行为异常

## 相关指标

- Bus Factor: 1
- 每周下载量: ~4700 万次（colors + faker 合计）
- 从倦怠表达到恶意行为: 约 14 个月
- 下游依赖数: 数千个直接依赖，数万间接依赖
