# left-pad 事件 —— Bus Factor = 1 的代价

## 背景

2016 年 3 月 22 日，npm 生态中一个名为 `left-pad` 的 11 行代码包被作者 Azer Koçulu 从 npm 仓库中删除。该包的功能极其简单：用空格填充字符串左侧至指定长度。

然而，这个看似微不足道的包被数千个项目直接或间接依赖，包括 Babel、React Native、Node.js 核心工具链等。删除后，全球大量 CI/CD 构建流水线瞬间中断，影响范围估计覆盖数百万开发者。

npm 最终不得不恢复该包的旧版本以恢复服务，这是 npm 历史上首次恢复已删除的包。

## 早期征兆

- **Bus Factor = 1**：整个包只有一名维护者（Azer），没有 co-maintainer，没有任何治理文档
- **零外部贡献**：GitHub 仓库几乎没有外部 PR，issue 无人响应
- **情绪化维护**：作者在删除前曾在 Twitter 公开与 npm 管理层发生冲突，情绪信号明显
- **许可证变更争议**：作者曾因 kik 公司要求其更改包名而公开表达不满

## 影响

- npm 全球构建中断约 2.5 小时
- 暴露了 npm 生态的深层脆弱性：微小依赖的级联影响
- 促使 npm 修改政策：禁止删除被大量依赖的包（改为 deprecated）
- 引发了"依赖地狱"的大讨论，许多团队开始审查依赖树深度

## 教训

1. **Bus Factor < 3 是高风险信号**：单个维护者情绪、健康、意愿的任何变化都可能导致项目消失
2. **微小依赖也需要尽调**：不要只看 stars 数，要看维护者数量和活跃度
3. **监控维护者情绪指标**：公开平台上的冲突、倦怠表达是预警信号
4. **间接依赖风险**：通过依赖树分析，识别哪些关键路径上存在单点维护者

## 相关指标

- Bus Factor: 1（极端危险）
- Stars: ~200（低知名度但高依赖）
- 最后更新到删除: 约 3 年无更新
- 下游依赖数: ~2500 个直接/间接依赖
