## 协作规范

### 1. 代码注释必须使用中文

所有代码中的注释、文档字符串（docstring）、README 说明，一律使用中文。这包括：
- Python 文件中的 `#` 注释和 `"""docstring"""`
- 配置文件中的注释
- API 文档中的描述文字
- 错误提示信息

### 2. 引入新技术栈前必须解释

每当项目需要使用一个新的技术栈、框架、工具或概念时，**必须先向用户解释它是什么、解决什么问题、在本项目中的作用**，再开始写相关代码。

解释应包含：
- **是什么**：用一句话概括
- **解决什么问题**：为什么需要它
- **本项目中的作用**：具体到哪个模块、什么场景下使用
- **类比（可选）**：用用户可能熟悉的技术做类比，帮助理解

例如引入 FastAPI 时，应先解释：
> FastAPI 是一个 Python 异步 Web 框架，类似 Java 的 Spring Boot。在本项目中负责提供 REST API 接口（如 `/api/v1/analyze`），接收用户提交的仓库地址并返回分析任务 ID。

### 3. 开发节奏

- 严格按照 `PROJECT_PLAN.md` 中的 **Phase 0 → Phase 1 → Phase 2** 顺序推进
- 每完成一个 Phase，应提醒用户当前进度和下一步计划
- 先让功能跑通，再考虑优化和扩展

### 4. 开发环境规范（必须遵守）

**不到所有开发完成，不碰 Docker 部署。**
- 当前阶段（Phase 2-3）：后端在宿主机 + venv 运行，Docker 只跑 db/redis
- Docker 部署是 Phase 5 的事情，开发阶段碰 Docker 只会拖慢迭代
- 不要因为"环境一致性"等理由提前引入 Docker 构建

**所有包安装必须在项目虚拟环境中进行。**
- venv 路径：`backend/venv/`
- Python 版本：3.12.10（不要用全局 Python 3.14）
- 安装命令：`backend/venv/Scripts/python.exe -m pip install xxx`
- 安装完要验证导入是否正常

### 5. 其他注意事项

- 用户是实习生水平，对 Agent、MCP、RAG 等概念可能不熟悉，讲解时由浅入深
- 项目用于简历，代码质量要过关，命名要规范
- 避免过度设计，优先实现规划中明确列出的功能
- 用户偏好可视化工具（DataGrip）查看数据库，而非命令行
- 用户希望分步推进、紧凑汇报，不要一股脑写大量代码
